第一章： 简介-什么是expect

expect是一种控制需要交互的应用的程序，这些程序交互式的给出提示框，并希望用户使用键盘输入作为回应。使用expect可以是这些人工操作自动化。使用自动化的交互程序，可以使你去解决以前没有考虑过的问题。

expect能够节省很多苦活的时间，这本书将会指引你开始学习expect之路，第一章节是一个全局概览。我将会提供一些简单的应用和简短的脚本，但是我不会完全的解释这些脚本。我希望能够勾起你的兴趣，刺激你学习的欲望，在之后的章节我会重新提到本章节中的概念，并且会有更加详细的解释。

fsck，一个unix的命令，用于检查文件系统， 可以使用-y或者-n的option参数来让shell脚本执行，手册中对-y的定义：


“假定对fsck的所有交互请求都返回一个yes的回应，这种使用方式必须很小心，因为即便服务器出现了错误，这还是一个免费的许可。”

-n的选项更加安全，但是大部分时间都没有使用。这种接口真的是无法原谅的差，并且很多的命令有着与此类似的风格，ftp，一个文件传输程序，有一个选项能够禁止程序的提示，以达到在shell上运行的效果。但是他没有提供一个在错误发生之后备用的动作。

expect是一种控制程序交互的工具，他解决了fsck的问题，通过非交互的方式提供了所有交互式的功能。expect并不是为了fsck设计的，但是他也能够解决ftp的错误。

ftp，fsck的例子例举了在shell的用户界面上运行脚本的主要限制。shell本身没有提供读程序的输出，写入程序的输入的方式。这意味着shell能够执行fsck但是同时必须失去程序的某一些重要的功能。而有一些程序压根就不无法被shell脚本直接执行，举例而言：passwd如果缺少用户交互式的的输入根本无法执行。相似的无法被自动化的程序还有telnet, crypt, su, rlogin, gdb。大量的程序都有这种依赖用户数据输入的毛病。

Expect 的设计的用途在于和需要用户交互的程序打交道。一个expect编程者能够写出描述对话框的脚本，然后Expect能够和需要交互的程序无交互的运行。控制命令可以通过脚本传给键盘，反之亦然。这些使expect能够去做那些重复性劳动的工作，而人类可以享受有趣的部分。

a very brief Overview.

expect 大部分时间都是用tcl写的，tcl是一种被广泛使用解释性语言。我们会在第二章快速的过一遍tcl。

当程序执行之后。expect等待三种返回值：welcome failed timeout ，并且解析返回值并做对应的动作。busy对应的动作是说明多个命令如何被解析的。至于timeout关键字是一种特殊的形式，他匹配的是一种保底情况，就是一段时间之后，没有其他返回值被匹配的情况下，匹配的动作。

Testing

假如无论有多少理由反对，你还是决定修改passwd的源码，在重新编译之后，你会希望测试这个程序，以保证这个程序在交互式的使用中完全正确，但是你会发现，你不能使用老的交互式的方法去测试新的程序，因为这种无交互式的程序是你修改源码的初衷。

测试交互式程序是否正确是expect另一个有用的地方。即使你从来没有想要自动化一个程序，但是你也会有测试的需求。passwd只是一个例子，另一个例子是你自己写的程序，可能你希望你的程序会对用户的键入产生响应。它可能会将先到的字母丢弃掉。或者在原始模式中需要一个终端，或者需要^C来激活信号又或者你需要去看排序的输出，去了解每一个接下来的命令。

程序在自动化之后逻辑会不同

有一些程序如何缓冲自己的输出取决于他们当前运行的方式。当程序在无交互的运行时，会自动的使用unix自带的I/O库中的缓冲他们的输出，因为理论上现在没人看输出结果，问题在于此时你如果想看程序的输出，但是又打算无交互的运行。expect能解决这个问题，他能使程序人为自己在有交互的情况下运行。

另一个例子，shell会强制用户输入控制字符（^Z^C）或者是关键字（fg,bg）等去切换当前执行的任务。但是这些并不能在shell脚本中使用。相似的，shell不能够在无交互的情况下直接运行那些单独为程序交互设计的功能比方说 history 等。就像之前passwd表现的那样。换句话说，你有可能构建一个shell脚本去测试特定的shell的功能，但是并不能严格的测试shell程序的方方面面。但是expect能够做到这一点，他能让shell程序使用交互功能赋予的程序控制特性，一个被expect触发的shell程序会认为自己以交互的方式运行，从而和往常一样控制任务切换等等。

little more about tcl

expect的语言设计适用性很广，他有控制结构也有数据结构，所以你能用他来做任何事。这可以弥补其他程序中缺乏的可编程性。 例如，大多数调试器不提供非常复杂的编程控制方法。 通常，您不能声明变量并且仅限于简单循环。

expect的核心功能是由tcl提供的，这是一种在其他工具中广泛使用的语言。tcl代表的是工具（Tool）命令（command）语言（language）的缩写。他作为一种库内置在应用中，同时提供了通用的语言特性。tcl解决了在程序语言设计中一个长久以来的问题。

事实上，大多数的工具没有非常复杂的内置的控制语言。shell代表了这种极致案例——他拥有非常丰富的语言甚至能直接启动其他程序。像.cshrc和.profile这些启动脚本可能写的非常复杂。没有其他程序拥有这种伸缩性。这也是一样，这些程序都有自己的语言，每次去掌握一个新的程序都令人畏惧。

在这两种极端中间的部分，诸如 ftp 这种受限制的特殊的语言，像在.netrc文件中被允许的不是很有灵活性但是好过于没有。很多类似的程序都和ftp设计的一样，受限，不能扩展，并且还不统一。

像是telnet之类的远端通信程序都没有控制语言，大部分的交互程序都想ftp和telnet一样，缺乏编程性。

这种程序编程性的缺乏可以理解。如果希望写一个控制打印器的程序，那么将扫描器、解析器、符号表等等都加入这个程序中并不划算。这整个程序不过才10kb。但是你无法预测未来这个程序将会被用在哪里。比方说你又有新的打印器。你的程序必须去理解这些。比方说你得将程序分享给其他人使用。正好他们有两种不同的打印器。可能他们又需要在某一天打印特殊的东西——在发薪日检查。这种需求是无穷无尽的。相比于每次都去定制化程序，还不如将原始的语言扩展。在最开始的时候使用通用的语言相当的有意义。

那些已经使用tcl的程序都设置好了。他们一开始就有很好的语言设定。但是很多我们被赋予的程序在tcl存在之前就存在了，而且大部分都不会被重写。那些程序已经停止开发了很久。（虽然可能因为钱，你能买到新版本）。幸运的是，expect允许你在交互程序上铺一层tcl，而不用去对源程序有任何改动。事实上，expect让那些无法被重新编程的程序可编程。

job control

正如你能同时和多个程序交互，expect也能在shell中通过fg和bg去切换当前处理的程序。expect也能改变shell的注意。当然也比你自己切换快很多。结果就是，expect可以成为不同程序之间的凝胶。使得那些本没有互相交互设计的程序协同工作。一个有意思的例子是由Ken Thompson 写的原始的国际象棋程序，设计成和一个用户交互，他提示移动棋子，并且以数据的代数变换回应用户输入。不幸的是他不允许自己的输出作为自己的输入。即便是他允许也没有一个管道同时处理同一个shell的输出和输入。

通过几行expect代码，就能使一个象棋线程和另一根象棋线程下棋，包括通信和包装输出以便作为另个程序的输入。我们会在第十章展示这个，并且也会有对这种技术有更正式的使用。

Background Process

expect 在自动化后台程序的时候能很有用，像是cron。通过cron不仅能自动化交互也能一开始的时候启动程序。如果你有一堆复杂需求，仅仅使用ftp是不够的。

甚至可以让cron进程在后台交互式联系您。 例如，ftp脚本可能需要密码才能继续执行其操作。 它可以为您搜索网络，然后向您询问。 或者你可以让它寻找一组用户。 获取密码后，expect将返回并使用它来完成任务。 通过让expect查询密码，您不需要将它们嵌入脚本中。

用cron的另一个常见的理由是做备份。

Using Expect On UNIX

expect可以和任何的unix程序交互，理论上expect可以做任何事，但是这并不是expect的本质，expect能包装任何你想包装的那种有着极差行为的程序，而外在行为和源程序一模一样。

理论上的操作

  • 一根线程可以通过成功的CAS操作，将_owner 的field 从none ->non-none；
  • 不变的条件: 一根线程在任一时刻，最多出现在一个 监视器锁的 列表内：CXQ, EntryList, WaitSet
  • 竞争锁的线程，通过CAS将自身push到CXQ队列，然后自旋/park()；
  • 通过 竞争之后，线程最终获取锁，他必须从CXQ或者EntryList脱离队列；
  • 正在退出的线程（The exiting thread 正在退出锁的线程） 在 entryList 上找出并unparks 一个 暂定的继承者线程。并且，正在退出的线程 并没有和 entryList上的继承者线程脱离关系。在unpark之后，wakee(被唤醒者，这里应该指的是继承者线程)将会重新竞争锁，最终 这个继承者线程会或者线程或者重新park自己。（获取到锁或者没有只能重新等待）。
  • 继承性 是由 一个竞争性的交接策略提供的。正在推出的线程 并不保证或者直接传递锁的所有权给继承线程，代替的是 正在推出的线程交出锁的所有权，并唤醒一个继承线程，继承线程（重新）竞争锁的所有权。如果EntryList为空但填充了cxq，则退出线程将把cxq排入EntryList。 它通过分离cxq（用CAS安装null）并将线程从cxq折叠到EntryList中来实现。 EntryList是双重链接的，而cxq是单链接的，因为用于排队最近到达的线程（RAT）的基于CAS的“推送”。
  • 并发不变性：
    1. 只有对象锁的拥有者才能接触或者操作 这个对象的EntyList，这种自有的排他的性质保护了并发冲突
    2. 只有对象锁的拥有者才能分离(detach) cxq
  • 监视器锁的entryList 的操作避免了锁，但是严格来说，这并不是无锁编程(lock-free), 进入队列的操作是无锁的，但是出列 并不是。
  • CQX 队列可以有多个并发的入队线程，但是只有一个分离线程，这个机制并不受ABA问题的影响，更确切的说，基于CAS的入队操作，并不关注ABA问题。
  • 总而言之，cxq和EntryList构成或形成了 由那些试图获取锁定的线程的单个逻辑队列。 我们使用两个不同的列表来提高 常数时间获取锁之后的出队操作 的几率（在:: enter（）结尾中）并减少列表末端的消耗。 （参见Michael Scott的“2Q”算法）。 一个关键的需求是 使保持监视器锁定时发生的队列和监视器元数据操作最少 - 也就是说，我们希望最小化监视器锁定保持时间。 请注意，即使少量的固定旋转也会大大减少EntryList | cxq上的 进出队列 操作的数量。 也就是说，旋转减轻了对“内部”锁的争用并监视元数据。
  • CQX是那些 最近才进入，并试图进入同步区（获取锁）的线程的集合。因为我们使用CAS的操作将线程放入 CQX ，所以RATs(recently arrived threads) 采用的是FILO的形式。我们在锁即将被释放并且CQX不为空，但是entryList为空的时候 将CQX排入到entryList中;
  • entryList 本身是很流行化的队列实现，问题是我们希望进出队列的动作可以能在固定时间常量时间内完成。如果我们需要一个优先级队列，那么类似于Solaris的sleepq就能很好地工作。 即，http：//agg.eng/ws/on10_nightly/source/usr/src/uts/common/os/sleepq.c。 队列规则在:: exit（）时强制执行，此时解锁线程将cxq排入EntryList，并相应地对EntryList上的线程进行排序或重新排序。
  • 除非 ‘直接获取了锁’（”lock barging”）, 这个机制能够提供公平的环形的排序队列。类似于 电梯扫描算法。1
  • 监视器同步子系统 避免使用了 原生的系统的同步(Synchronization)(可能是一个lock指令的意思吧) 除了 狭义的平台指定的 park-unpark 抽象。在 os_solaris.cpp 中有提到park、unpark的语义。换句话说这个monitor的实现只用到了原子操作和线程的park、unpark。monitor子系统对线程的 RUNNING->BLOCKED和 BLOCKED->READY这些线程状态改变的管理通过底层 操作系统 READY<—>RUN 的转变。
  • 处于等待状态的线程在wait-set之中， wait()方法将方法的调用者放入wait-set中。
  • notify() 或者 notifyall() 简单的将waitset中的线程放入 CXQ或者entryList中。随后，exit()操作将unpark() 被唤醒者。使用notify()唤醒线程unpark线程是一项没有效率的动作。很有可能被唤醒者将被钉在唤醒线程持有锁上。

这是一篇笔记

关于娱乐

基于本书所述的各种措施，每天理论上能多出2个小时的时间，（虽然我还没省出这么多时间来），这两个小时的时间很有可能都是在被动性娱乐中被消耗，先不谈被动性娱乐的概念，书中所述被动性娱乐大概就是电视之类，比之现在我的处境，大概就是斗鱼TV了吧，通过视觉输入，各种信息纷至沓来，注意力不断的被分散，无法集中，跟着情节或者剧情不断推进，欲罢不能直至睡觉时间到——这听上去其实不错——但是，这很被动，这里的被动的意思是，我们的意识或者是此一刻的享受我们是被动的，我们没有主动参与——听上去我们是不是该给主播刷点礼物或者带点节奏？——当然不是，我的理解是，这之所以被动，最根本的原因是没有目标，只是放松，这种放松的方式当然没有问题，但是需要控制时间。并且对提高专注力没有提高。

主动性的娱乐

相对应的有被动性的娱乐也有主动性的娱乐，看书，象棋，围棋这些，在繁忙的一天工作之后能够继续下象棋或者围棋，这个有点天方夜谭，但是在以往的经验中，在晚上9点到10点之间全神贯注并且状态十足的下3盘象棋是完全没问题的。看书的话，理论上不会去看一些十分消耗脑力的书籍，文学性的书籍maybe ok。当然提高一些工作技巧的书籍更好，比方说 《思维导图的使用》《思维导图的设计逻辑》，或者是我正在践行的时间管理的书籍。下象棋或者围棋我不知道能够有什么输出？或者只是单纯的锻炼脑力。如果是看书的话，最好能够有所输出，比方说这篇文章，在看完时间管理的书籍之后，能够对这本书的一些概念进行整理和输出。

  1. 定制目标，比方说看书，读后感，针对当前的情况作出一些改变。这会比较难吧，我也正在学习。
  2. 有所记录，很多时候灵光乍现，灵感转瞬即逝。需要一些东西记录。史航的读书方法也在践行中。
  3. 有所输出，记录就是输出这种说法相当错误，学习就是不断的总结，归纳，反省。

如何减少被动性的娱乐

此处的标题更像是一个措施。书中给出的建议是，把想要看的电视录下来，回头看。这无疑是一个好方法，但是在现实生活中，无法和周围的朋友聊同一个昨天播出的节目的八卦，多么残酷的一件事，当然没有这种白痴朋友，麻烦瞬间就解决了。回到现实，比方说是美剧，现在的网络技术，可以一次性下载，找相对空闲的时候观看。这个措施真的不错。

心流

这这篇文章中有一个term: 心流 ，说实话这个中文单词真的很中二，这个文章的作者又是一个日本人，我联想到了 浪客剑心，一刀流，绯村拔刀斋等等。但是这的确是一个西方科学家提出的术语：flow或者是Zone——翻译的确信达雅——一个人进入忘我的状态，人与外界的挑战有110byte/s，此时没有任何其他的注意力来关注其他的事情。这种状态在东方有很多名字：禅（zen）、虚、化境、忘境。这里可以看一看打坐或者是冥想在这里的说法。
